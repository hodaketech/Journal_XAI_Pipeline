# LLM-Assisted Verbalization Integration Plan

## Purpose

This plan addresses the reviewer comment:

> An LLM was used, but there is no information of how it is being used and no information about the parameters (e.g., model, temperature, technique, etc.). This makes it hard to reproduce the results.

Repository inspection confirms that the current RL source code does **not** contain an LLM implementation, OpenAI/ChatGPT API call, prompt file, model name, temperature/top-p setting, or generation script. The repository does contain deterministic/template-style explanation rendering in validation scripts. The correct integration is therefore an **LLM-assisted verbalization layer** in the Explain stage only.

The LLM must not participate in:

- RL training.
- Policy learning.
- Reward decomposition.
- Action scoring.
- Action selection.
- RDX/MSX/BTMM/RWS computation.
- Quantitative metric evaluation.

The LLM may participate only after Decompose / Score / Select have produced structured numerical evidence.

## Proposed Integration Architecture

Do not modify the core RL code. Add a revision/post-processing layer:

```text
existing experiment outputs
    -> build_explanation_records.py
    -> explanation_record_schema.json validation
    -> prompt_template.md
    -> offline ChatGPT or API LLM verbalization
    -> llm_outputs.jsonl
    -> validate_verbalized_explanations.py
    -> validated explanations for manuscript/table examples
```

Recommended files:

- `revision_outputs/explanation_record_schema.json`: schema for normalized records.
- `revision_outputs/sample_explanation_records.jsonl`: traceable examples from existing outputs.
- `revision_outputs/prompt_template.md`: constrained prompt for future LLM-assisted rendering.
- `revision_outputs/llm_validation_checks.md`: validation protocol for LLM outputs.
- Optional future script: `tools/build_explanation_records.py`.
- Optional future script: `tools/validate_verbalized_explanations.py`.

Implemented in this revision:

- `tools/build_explanation_records.py`
  - Reads the checked-in Maze and Connect6 outputs.
  - Builds normalized JSONL records following `revision_outputs/explanation_record_schema.json`.
  - Default output: `revision_outputs/explanation_records.jsonl`.
  - Example command:

```bash
python tools/build_explanation_records.py
```

- `tools/validate_verbalized_explanations.py`
  - Validates an LLM output JSONL against the source evidence records.
  - If no LLM output file is supplied, it validates the existing template explanations stored in the records.
  - Default report: `revision_outputs/llm_validation_report.json`.
  - Example commands:

```bash
python tools/validate_verbalized_explanations.py --records revision_outputs/explanation_records.jsonl
python tools/validate_verbalized_explanations.py --records revision_outputs/explanation_records.jsonl --llm-outputs revision_outputs/llm_outputs.jsonl
```

Current implementation test:

- `tools/build_explanation_records.py` generated `601` records:
  - 100 records from Case Study 1.
  - 100 records from Case Study 2.
  - 401 records from Case Study 3.
- `tools/validate_verbalized_explanations.py` checked the existing template explanations and reported:
  - 529 passed.
  - 72 failed.

The failures should not be interpreted as LLM failures, because no LLM output has been generated yet. They indicate that some existing template explanations are not fully aligned with the new strict schema checks, for example missing the compared alternative action or having a mismatch between validation text and workbook evidence. This is useful for revision because it shows why a constrained LLM prompt plus validation layer is needed.

## Verbalization Protocol

1. Read evidence from existing logs/workbooks.
2. Convert each row to the unified explanation record schema.
3. Save JSONL records with source file and row id.
4. Send records to the LLM only for natural-language rendering.
5. Require a structured JSON response with `local_explanation`, `evidence_used`, and `limitations`.
6. Validate LLM output against the source record.
7. Use only validated outputs in the manuscript.

## Replacement Subsection for Section 5.4

### Explain: structured evidence and optional LLM-assisted verbalization

In the Explain stage, the implemented XRL pipeline first produces a structured explanation record from numerical decision evidence. For Maze, this record includes the state coordinates, selected and alternative actions, decomposed Q-value margins, RDX values, MSX positive and negative contributors, and the total decision margin. For Connect6, the record includes the board/move descriptor, BTMM/RWS scores or gamma values, selected move probabilities, and phase-level context when available.

The LLM is used only as an optional offline verbalization layer. It does not participate in policy learning, reward decomposition, action scoring, action selection, explanation selection, or quantitative evaluation. Its input is the structured explanation record, and its output is a concise human-readable explanation following a fixed schema: selected action, compared alternative, top positive contributors, top negative contributors if present, stochasticity note when relevant, and a short takeaway.

To improve reproducibility, each verbalized explanation is linked to its source file and row id. The generated text is then checked by the Validate stage against the original structured record. The validation checks whether the selected action and alternative action are preserved, contributors are not inverted, numerical signs and key values are retained, unsupported contributors are not introduced, and stochastic selection is described correctly for epsilon-greedy or RWS cases.

The released repository contains the numerical evidence and deterministic template-based explanation renderers. The original ChatGPT session metadata for the manuscript examples was not retained in the repository; therefore, the revised manuscript treats those prose examples as offline verbalizations of reproducible structured evidence rather than as part of the implemented RL algorithm.
